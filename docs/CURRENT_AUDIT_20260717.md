# Earth on Minecraft 当前复盘与下一阶段计划

日期：2026-07-17
目标平台：Minecraft 26.2 / NeoForge 26.2.0.7-beta / Java 25
当前源码版本：0.1.10 开发中测试版

## 1. 本轮审查边界

本轮只审查 `Earth on Minecraft / 我的地球` 本体。`Earth Human`、`Earth Online: Magic`、`Earth Online: Xuanhuan` 和 `Temple of Nihility` 仅用于提取已经总结的工程经验、玩法边界和联动契约，不修改它们的代码。

本体定位保持不变：复刻现实世界现代及现代以前的常见内容，但不复刻现实难度；保留 Minecraft 的物品生态、堆叠、合成魔法和兼容出口。

## 2. 跨窗口经验

其他窗口最近的复盘有六条可以直接用于本体：

1. 先建立实现真值表，再写手册、路线图和发布说明，不能把讨论中的设想写成已实现功能。
2. 内容数量不是闭环。玩家必须能找到入口、理解当前状态、知道下一步，并得到有用途的最终产物。
3. 不同体系和设备不能只换名称、颜色和图标。视觉身份、槽位、状态、操作节奏和反馈都应对应真实职责。
4. 多方块图、投影、手册和验证逻辑应来自同一份结构定义，不能分别手写并逐渐漂移。
5. 高清源图不是验收结果。必须检查最终资源尺寸、实际 GUI 尺寸、双语文本、动态数值、最小分辨率和游戏内状态。
6. 联动数据必须有所有者。优先使用标签、数据配方、持久化契约和小型只读 API；可选模块缺失时必须安全退化。

### 3.2 构建基线

本轮实际执行：

```text
python tools\validate_resources.py
OK json=1005 png=338 models=658 refs=344 blocks=52 items=241
integrationTags=15 worldgenRefs=35 worldgenHeights=31 strata=4
vanillaOreSuppression=20 playerText=9 releaseNotes=1

.\gradlew.bat build --no-daemon --offline

```

`git diff --check` 没有补丁错误，只有现有 Windows 行尾转换警告。Gradle 仍报告 26.2 使用最新已知 26.1-snapshot 能力映射和少量废弃 API 警告。

### 3.3 不能算作已完成的部分

1. 最新多方块和 UI 改动尚无对应客户端实机截图、进世界日志和交互验收。
2. Gradle 测试任务为 `NO-SOURCE`，369 条路线、结构旋转、旧存档迁移和电网行为没有自动测试。
3. `earth/rocks` 与 `earth/strata` JSON 当前通过静态校验，但没有发现运行时资源加载器；它们还不是驱动游戏逻辑的单一真源。
4. 369 条处理路线硬编码在 `ProcessingMachineBlock.createRecipes()`；JEI、tooltip 和手册虽能读取部分运行时对象，但内容维护仍高度集中在 Java。
5. 手册约 857 行、81 个页面构造点，主要是静态 Java 文本；新增路线后容易与实际配方、功率和结构漂移。
6. 21 台机器共用一个 `ProcessingMachineMenu/Screen`。当前已显示机器图标、工艺族、动力和状态，但核心槽位与操作骨架仍高度相同。
7. `data/earth_human/.../hydration.json` 是兼容数据包，不是本体玩法。它合理但必须继续明确所有权，并确保 Earth Human 缺失时安全加载。

## 4. 当前主要问题

### P0：发布阻断项

#### 4.1 控制面板与多方块几何不一致

结构坐标已经把控制面板放到正面底层中心，但 `ControlPanelBlock` 的碰撞和两个模型仍是 2 像素厚的贴面薄板。需要统一为一种明确规则：

- 控制面板占用完整结构格，并与正面机壳齐平。
- 未连接、已连接待机、已连接运行三种状态使用相同几何。
- 隐藏在结构内的控制器必须把运行状态传递给可见面板和成型外壳。
- 右键任意有效成型部件都能找到并打开控制器。
- 投影保持无碰撞、不可选中、不可破坏，并由实际结构坐标生成。

#### 4.2 最新运行时基线缺失

修正控制面板后必须在真实 26.2 NeoForge 实例完成：

- 新世界创建与进入。
- 八种结构的四向投影、成型、拆除和相邻结构测试。
- 控制面板三状态、任意结构部件打开、接口和输送带连通。
- 处理机、发电机、电池箱、手册和 JEI 的实际截图。
- `latest.log` / `debug.log` 中无注册表、数据包、模型和客户端类加载错误。

#### 4.3 建立可复现 Git 基线

当前 Git HEAD 为 0.1.9 发布基线，工作树包含 0.1.10 基础 GUI 与交互检修。完成构建、部署和测试版发布后，应先提交这一可复现基线，再继续扩展内容。

## 5. 下一阶段执行计划

### 当前优先级调整：Living World

用户已将生物、居民和聚落确定为下一条大型主线。完成 P0 的短稳定化门槛后，优先进入居民身份、职业、交易、掉落、村庄增强、资源镇、工业镇和机器品质升级。详细方案见 [生物、聚落与品质升级计划](living-world-settlement-plan.md)。

数据驱动、机器界面分化和现有工业闭环继续作为这条主线的支撑工程，但不再要求全部完成后才开始生物与聚落开发。

### P1：建立单一数据源

先停止继续向 `createRecipes()` 和静态手册堆路线，建立运行时数据模型：

```text
data/earth_on_minecraft/processing_routes/*.json
```

第一版字段至少包括：

- 机器类型和工艺族。
- 物品或标签输入；预留多个输入。
- 多个确定产物与可选概率产物。
- 时间、功率、热源/电网模式、温度或压力提示。
- 催化剂与容器返还语义。
- 手册章节、现实说明、JEI 排序和可选命名空间条件。

机器执行、JEI、tooltip、手册和校验器必须读取同一份路线数据。先迁移铁、铜、石油、水处理、电池五条代表链，验证后再批量迁移其余路线。

地质数据同步处理：要么实现 `rocks/strata/deposits` 的运行时加载与查询 API，要么明确这些 JSON 只是资料文件，不能继续把它们写成已驱动世界生成。

### P2：机器交互与界面分化

不再把 21 台设备理解成同一个熔炉换皮。采用“工艺族基础布局 + 设备专属控件”的结构：

| 工艺族 | 应有的交互差异 |
|---|---|
| 破碎/磨矿 | 给料、粒度、磨矿介质或转速、磨损/堵料反馈 |
| 筛分/磁选 | 多级产物流、分选阈值、旁路与回收率提示 |
| 浮选/浸出 | 主料、介质/药剂、液位或浸出条件、尾矿流 |
| 热工 | 燃料/电热、温度区间、烟气/渣出口、升温状态 |
| 电解 | 电极、电解质、功率密度、阳极/阴极产物 |
| 反应/聚合 | 多输入、催化剂、温度/压力、批次状态 |
| 塔器 | 进料位置、回流/吸收介质、顶部/底部产品 |
| 成型 | 模具、压力、批量与产品规格 |

界面仍保持 Minecraft 原版容器的尺寸感、方框和可读性，不做网游面板。每台机器至少有独立的过程示意、状态反馈和悬浮说明；不能只换标题、图标和颜色。

同时升级静态预览：使用真实 GUI 尺寸、实际语言文本、最长动态数值、物品图标、按钮和 tooltip 热区生成联系表，并检查小窗口和低分辨率。

### P3：现有玩法闭环

按玩家路线而不是按物品数量审查：

1. 第一块铁、第一块铜、第一台发电机、第一段电缆、第一台自动机器。
2. 铁、铜、铝、钢、塑料、橡胶、玻璃、陶瓷、水泥的主要用途。
3. 核燃料链是否真正接入可运行核热设备，而不只是中间物。
4. PLC、传感器、执行器、机器人臂和检测模块是否进入实际自动化功能，而不只是合成终点。
5. 尾粉、矿渣、污泥、废酸碱、废塑料和乏燃料是否有清晰回收或稳定化出口。
6. 输送带增加缓冲、分流、过滤、转弯和阻塞诊断；接口保持可见单格货物运输语义。
7. 电网补齐变压、开关、容量、线路等级和诊断；避免三种电缆只有外观粗细不同。

每条路线验收时必须回答：如何发现、第一步是什么、需要哪台设备、为什么这样处理、产生什么副产物、最终能做什么。

### P4：多方块与美术统一

1. 控制面板、输入/输出接口、管线和机壳先完成基础质量统一。
2. 每种结构使用独立的现实设备原型和可读外形；现实没有固定原型时采用清楚的 2x2、2x3、3x3 或塔式长方体。
3. 成型后不只是方块换贴图，应形成连续壳体、贯通管线、支撑、视窗、平台或炉体；短期用结构角色模型，中期再评估 multipart/BER。
4. 运行/待机使用同源贴图和相同几何，状态变化只改变有意义的灯、热区、液位或运动部件。
5. 正面、侧面、顶部和背面遵循同一物理外壳，禁止把内部剖面直接贴在方块表面。
6. `image-2` 只在供应端恢复后后台生成；先做代表样张和最终 256x 联系表，再批量替换。

### P5：本体的非工业现实内容

工业是当前最完整的体系，但本体不能只剩工业机器。完成 P0-P3 后，按可玩闭环补充这些现实地球内容：

- 地质与水文工具：岩芯、样品分析、含水层/盐卤/地层查询、矿床勘探，不把信息藏在调试命令里。
- 土壤和农业基础设施：土壤改良、灌溉、排水、肥料使用和盐碱修复；不引入人体营养或食物腐败。
- 公共工程：道路、沥青、混凝土、玻璃、陶瓷、保温、防水、给排水和照明的实际方块用途。
- 能源设施：风、光、水、核热和储能形成不同现实特征的功能路线，不只增加材料名称。
- 测量与控制：温度、压力、流量、料位和质量检测成为机器反馈与自动化条件。
- 回收与城市物质流：金属废料、塑料分类、电子废料、纸纤维和建筑废料回到既有工业链。

人物属性、疾病、食物保存继续属于 `Earth Human`；修仙、魔法和远未来科技继续属于独立项目。

### P6：生态联动合同

本体继续只提供现实材料、环境事实和查询出口：

- 保持现有 15 个角色标签和可选依赖安全退化。
- 新增有界、可缓存、只读的 `GeologyQueryApi`，供附属查询主要岩性、矿床角色和工业环境摘要。
- 本体手册只显示“当前安装组合增加了哪些材料出口”，不复制附属玩法教程。
- 跨命名空间数据文件必须在文档和校验器中标记为兼容包，并由目标模块公开标签语义。
- 发布前执行四个项目 15 种非空安装组合的专服冒烟测试；重点组合再做客户端交互验证。

## 6. 固定执行顺序

1. 修正控制面板完整方块几何和三状态传播。
2. 构建、部署、客户端实机检查八种多方块、全部界面和日志。
3. 提交 0.1.10 当前可复现基线。
4. 建立聚落档案、居民档案、姓名池、角色数据和聚落公告板。
5. 实现首批六个现实职业、聚落交易、居民/掠夺者掉落和工作行为。
6. 增强原版村庄，再实现矿业镇和电气化工业镇。
7. 为五台代表机器和首批居民建立生产质量动作、声音和特效标准。
8. 并行推进处理路线数据源、机器工艺族界面和现有工业用途闭环。
9. 补齐铁/钢管网、阀门、泵、配电、仓储和城镇物流。
10. 最后推进聚落发展、地质查询 API 与四项目动态联动页。

在第 1 至 3 步完成前，不继续新增大批物品或机器。Living World 第一轮必须按完整纵向切片实施，不做几十个同质职业、空壳建筑或只换颜色的动作。
